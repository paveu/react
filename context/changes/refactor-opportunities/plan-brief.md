# Host-Config-Seam Refactor Opportunities — Plan Brief

> Full plan: `context/changes/refactor-opportunities/plan.md`
> Research: `context/changes/refactor-opportunities/research.md`

## What & Why

The host-config seam carries documented tech debt (research items T1–T7). This plan executes the three lowest-cost, highest-leverage of them — **T4** (a runtime fork-parity test), **T2** (codegen for the hand-maintained `.custom`/`.noop` passthrough forks), and **T1** (extract the triplicated fork-selector logic). The aim isn't to fix the deepest debt but the cheapest, reversible guards that reduce the pain of the rest — all confined to build-time code with zero `packages/` runtime changes.

## Starting Point

Each renderer gets a `ReactFiberConfig` fork swapped in at build time by three toolchains (rollup/jest/flow). `.custom.js` holds 166 hand-written `$$$config` passthroughs (de-facto canonical surface); `.noop.js` mirrors them plus a leading `export *`. Flow is the *only* guard that a fork didn't drop a symbol; `dependency-cruiser` exists but isn't wired into CI. The longest-prefix fork selector is reimplemented in all three toolchains (no drift yet, but latent).

## Desired End State

A Jest parity test asserts all 7 forks expose the same symbol set (derived from `.custom.js`) and runs in `yarn test`. `.custom.js`/`.noop.js` are generated byte-identically from that canonical set via `yarn generate-host-config-forks`, kept fresh by a CI check. The selector logic lives in one `scripts/shared/resolveHostConfigFork.js` that all three toolchains import.

## Scope

**In scope:** parity test (T4), `.custom`/`.noop` codegen + CI freshness (T2), shared selector extraction + 3 call-site swaps (T1).

**Out of scope:** T5 (controlled-components knot — concept redesign), T6 (DOM↔reconciler cycle — guard not rebuild), T7 (contract segmentation — only if pain persists), T3 (DOM config unit test), any `packages/` runtime change.

## Architecture / Approach

A single parser helper derives the canonical ordered symbol set from `.custom.js` (Babel parse — handles the multiline `$$$config` wrapping that makes naive grep undercount 135 vs 166). Both the T4 test and the T2 generator consume it, so they agree by construction. Codegen mirrors the existing `createFlowConfigs.js` / `generate-inline-fizz-runtime.js` patterns. T1 extracts the `findNearestExistingForkFile` longest-prefix algorithm to a pure shared module; the three toolchains keep their own I/O.

## Phases at a Glance

| Phase | What it delivers | Key risk |
| --- | --- | --- |
| 1. T4 parity test | Runtime net asserting 7-fork export parity | Allowlist mis-scoped; multiline parse undercount |
| 2. T2 codegen | Generated byte-identical `.custom`/`.noop` + CI freshness | Generator can't match exact formatting/order → diff fails |
| 3. T1 selector | One shared selector, 3 call-sites swapped | Subtle behavior drift in one toolchain's resolution |

**Prerequisites:** Phase 2 depends on Phase 1 (the net). Phase 3 is independent. Node + `@babel/parser` (already deps).
**Estimated effort:** ~3 sessions, one PR per phase.

## Open Risks & Assumptions

- Whether the React team accepts a runtime parity test, given they've favored tightening the Flow gate (#35944) — open question for review.
- Whether codegen output can be byte-identical without changing `$$$config` formatting — proven only by the diff gate.
- Whether T7 pain actually disappears after T4+T2 — measurable only afterward.

## Success Criteria (Summary)

- `yarn test` runs a parity test that fails if any fork drops/adds a symbol; `yarn flow` unchanged.
- Regenerating `.custom`/`.noop` produces zero diff; a CI step keeps them fresh.
- All three toolchains resolve forks via one shared, unit-tested selector.
